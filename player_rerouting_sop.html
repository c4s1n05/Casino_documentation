SOP — Player Re-routing After 2 Failed Deposits

Defines the automated and manual response when a player experiences two consecutive acquirer-declined deposit attempts within a single session. Covers re-routing logic via SoftSwiss, on-site messaging standards, Customer Support scripts and CRM recovery actions.

📅 May 2026
👤 Owner: Payments Team
🔗 Platform: SoftSwiss
⚡ Trigger: 2 acquirer declines in session
💡
Why This Matters

A player who fails to deposit twice in a session is at extremely high risk of abandonment. Research across iGaming shows that conversion drops by over 60% after a second consecutive failure. The goal of this SOP is to remove friction as fast as possible — through re-routing, clear messaging and proactive support — before the player gives up and churns. Every second of delay after the second failure costs money.

01
Trigger Logic — When This SOP Applies

This SOP is activated specifically when failures are attributed to acquirer decline codes — not player-side errors (wrong card number, insufficient funds, expired card). Distinguishing between these is critical to avoid incorrect re-routing.

Code 12 — Invalid Transaction (×2)Acquirer not accepting this merchant type — rerouting to different acquirer may resolve✅ Yes — re-route

Code 51 — Insufficient Funds (×2)Genuine player balance issue — re-routing will not fix this❌ No — suggest lower amount or alternative method

Code 82 — Incorrect CVV (×2)CVV entered incorrectly — player-side error❌ No — prompt player to check card security code
⚠️
Session Definition

Two failures count as in-session if they occur within 30 minutes of each other on the same payment method. If the player changes payment method between attempts (e.g. switches from card to e-wallet), the counter resets. If the player returns after 30 minutes, they are treated as a new session. The session window can be adjusted by the Payments Lead in the SoftSwiss configuration.

02
Player Journey — End to End
1
First Deposit Attempt — Declined (Acquirer)
Player submits deposit. SoftSwiss processes via primary acquirer. Acquirer returns decline code (e.g. Code 05 or Code 91). SoftSwiss logs the decline. Player sees generic error message on deposit page.
System: SoftSwiss logs decline code + timestamp
2
First Failure — On-Site Message Displayed (Attempt 1)
After first acquirer decline, the deposit page displays a soft notification: reassures the player the card was not charged, acknowledges the failure, and offers simple troubleshooting. Does not yet suggest alternative methods to avoid alarming the player prematurely.
Player sees: Attempt 1 message (see Section 4)
2
Second Deposit Attempt — Declined (Acquirer)
Player tries again on the same payment method. SoftSwiss processes via same or different acquirer depending on retry logic. Second acquirer decline returned. SoftSwiss re-routing trigger fires (if configured — see Section 3). Two-failure threshold reached.
System: 2-failure threshold triggered → re-routing logic activates
!
🚨 Trigger Point — Re-routing & Messaging Activated
At this exact moment: (1) SoftSwiss attempts to re-route subsequent attempts to a backup acquirer. (2) On-site messaging upgrades to the two-failure message — clear, helpful, proactive. (3) CS alert fires in Telegram with player session reference if chat is available. (4) Player email capture triggered for CRM recovery if player abandons.
System auto-actions on 2nd decline Player sees: Attempt 2 message — alternative methods presented CS: Telegram alert fires with session ref
→
Player Path A — Rerouted Card Attempt Succeeds
SoftSwiss re-routes to backup acquirer. Player tries again. Deposit approved. Player continues session. System logs successful re-route for reporting. No further action required beyond logging.
✅ Resolved via re-routing
→
Player Path B — Switches to Alternative Method
Player selects an alternative method (e-wallet, crypto, bank transfer) prompted by on-site messaging. Deposit succeeds via alternative method. System logs method switch for CRM analysis. Payment Team reviews why card route failed and requests SoftSwiss fix.
✅ Resolved via method switch Log: method switch event for CRM reporting
?
Player Path C — Third Failure or Abandonment
Player tries re-routed card and fails a third time, or player exits without depositing. Third failure: SoftSwiss escalation required immediately. Abandonment: CRM recovery flow activates within 15 minutes if email is on file — targeted re-engagement with alternative methods highlighted.
CS: Proactive outreach if player has live chat open SoftSwiss: Escalate immediately on 3rd failure
03
Re-routing Logic via SoftSwiss
🔀
Re-routing is Controlled by SoftSwiss

The brand does not control acquirer routing directly. All re-routing happens through SoftSwiss configuration. The Payments Team must work with the SoftSwiss account manager to set up automated re-routing rules in advance — so that when the 2-failure trigger fires, re-routing happens automatically without requiring a manual call each time.

Re-routing RuleConfiguration Request to SoftSwissExpected Outcome
AUTOMATED RE-ROUTING (Configure in advance with SoftSwiss AM)

BIN-specific routing ruleRequest SoftSwiss configure: if a specific BIN prefix has >50% decline rate on primary acquirer, route all transactions from that BIN to secondary acquirer automaticallyReduces failure rate for known problematic BINs before players even experience two failures
Time-of-day routing ruleIf certain acquirers have higher success rates at specific times (e.g. overnight), request SoftSwiss configure time-weighted routing to prefer the most reliable acquirer for the current time windowImproves overall success rate during known low-performance windows
Card scheme routing preferenceRequest SoftSwiss configure separate routing logic for Visa vs Mastercard — if one scheme performs better on a specific acquirer, route that scheme accordinglyOptimises approval rate by card type rather than treating all cards identically

Emergency reroute on live incidentIf automated rules have not resolved two-failure spike, Payments Lead contacts SoftSwiss AM to manually switch routing to backup acquirer immediatelyFaster than waiting for automated rules to cycle; SoftSwiss can make routing changes in the back office in real time
Temporary route suspensionIf a specific acquirer is failing broadly, request SoftSwiss suspend that acquirer temporarily and route all traffic to backup. Restore once acquirer is confirmed stable.Protects success rate at the cost of redundancy — backup acquirer handles full load temporarily

07
CRM Recovery — Abandoned Deposit Players

Players who abandon after two failed deposits must be re-engaged within 15 minutes via CRM if email or push notification is available. The message must acknowledge the issue without alarm, offer a clear alternative, and optionally include a soft incentive to reduce the friction of returning.

Quarterly Optimisation Review

Every quarter, the Payments Team and Marketing Manager should review the on-site messaging A/B test results, CRM recovery rates and re-routing success data. Present findings and recommendations to the COO in the quarterly business review.
